# 🛗 Smart Elevator Control System – ER Diagram Explanation

## 🏢 Buildings

* Represents each building connected to the platform.
* Stores basic details like name and address.
* A building acts as the top-level entity containing:
  * floors
  * elevators
  * shafts
  * requests

---

## 🏬 Floors

* Represents floors inside a building.
* Each floor:
  * belongs to one **building**
  * has a **floor number** and optional label (e.g., Basement, Ground)
* Used as:
  * source and destination for ride requests
  * reference point for elevator movement

---

## 🛗 Elevator Shafts

* Represents physical shafts inside a building.
* Each shaft:
  * belongs to one **building**
  * contains exactly one **elevator**
* Helps model real-world constraints where elevators operate within shafts.

---

## 🚡 Elevators

* Represents individual elevator units.
* Each elevator:
  * belongs to a **building**
  * is assigned to a **shaft**
  * has capacity and installation details
* Does NOT store dynamic data like rides or status (kept separate).

---

## 🔗 Elevator–Floor Mapping

* Defines which floors an elevator can serve.
* Many-to-many relationship:
  * one elevator → many floors
  * one floor → many elevators
* Enables:
  * zoned access
  * restricted floor servicing

---

## 📢 Floor Requests

* Represents ride requests generated by users.
* Each request includes:
  * source floor
  * destination floor
  * direction (UP/DOWN)
  * status (pending, assigned, completed, cancelled)
* Acts as the starting point of the elevator workflow.

---

## 🎯 Ride Assignments

* Maps a request to an elevator.
* Each request:
  * is assigned to exactly one **elevator**
* Separates:
  * request generation
  * dispatch decision
* Enables tracking of which elevator handled which request.

---

## 🚕 Ride Logs

* Stores completed ride data.
* Each ride:
  * is linked to one **elevator**
  * originates from one **request**
* Captures:
  * start floor
  * end floor
  * timing and duration
* Used for:
  * analytics
  * performance tracking

---

## 🧭 Ride Stops

* Represents intermediate stops during a ride.
* A single ride can include multiple stops:
  * pickup floors
  * drop-off floors
* Each stop stores:
  * order of stop
  * arrival and departure times
* Provides realistic modeling of elevator movement.

---

## 📡 Elevator Status Logs

* Tracks real-time and historical elevator states.
* Each record includes:
  * current status (idle, moving, maintenance, offline)
  * current floor
  * timestamp
* Stored separately to:
  * enable time-series tracking
  * avoid overwriting current state

---

## 🛠️ Maintenance Records

* Stores maintenance history for each elevator.
* Includes:
  * type (inspection, repair, emergency)
  * status (scheduled, in-progress, completed)
  * timestamps
* Ensures:
  * maintenance does not overwrite operational data
  * historical tracking is preserved

---

## 🔗 Relationships Summary

* One **building** → many **floors**
* One **building** → many **elevator shafts**
* One **building** → many **elevators**
* One **shaft** → one **elevator**
* Many **elevators ↔ many floors** (via mapping table)
* One **floor** → many **requests**
* One **request** → one **ride assignment**
* One **elevator** → many **assignments**
* One **elevator** → many **ride logs**
* One **ride log** → many **ride stops**
* One **elevator** → many **status logs**
* One **elevator** → many **maintenance records**

---

## 🧠 Key Design Points

### ✅ Separation of Static and Dynamic Data

* Static:
  * buildings, floors, elevators
* Dynamic:
  * requests, assignments, rides, status logs
* Prevents data bloating and improves scalability.

---

### ✅ Real-World Elevator Behavior

* Supports:
  * multiple elevators per building
  * multiple floors per elevator
  * multi-stop rides
* Accurately reflects real infrastructure systems.

---

### ✅ Request Lifecycle Modeling

* Flow:
  * request → assignment → ride → logs
* Ensures traceability across the system.

---

### ✅ Time-Based Tracking

* Status and maintenance are logged separately.
* Enables:
  * historical analysis
  * uptime monitoring
  * debugging

---

### ✅ Many-to-Many Handling

* Elevator–floor mapping handled via junction table.
* Supports:
  * flexible servicing
  * zoned elevator systems

---

### ✅ Ride Analytics Ready

* Ride logs + stops allow:
  * usage tracking
  * performance metrics
  * optimization insights

---

### ✅ Maintenance Safety

* Maintenance records are independent.
* Elevators can be serviced without losing historical ride data.

---

## 🚀 Overall Design Strength

* Scalable for multi-building infrastructure systems  
* Supports real-time request handling and tracking  
* Proper normalization and clean separation of concerns  
* Realistic modeling of elevator operations  
* Ready for analytics and future optimization systems
